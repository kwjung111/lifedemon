# Telegram 메시지 해석 설계

Life Daemon은 자유문장을 기능별 정규식으로 흉내 내지 않습니다. Telegram 자유문장 하나를 전역 해석기가 정확히 한 번 구조화하고, 각 모듈은 그 결과를 검증한 뒤 실행합니다.

## 한 메시지의 처리 순서

1. 런타임이 개인 채팅 ID와 사용자 ID를 확인합니다.
2. 고정 명령어와 버튼 callback이면 AI 없이 정해진 프로토콜로 처리합니다.
3. 그 밖의 자유문장은 답장한 메시지의 공개 항목 정보와 함께 전역 AI 해석기로 보냅니다.
4. 해석기는 `route`, `domain`, 대상 번호, 신뢰도와 기능별 필드를 JSON 스키마에 맞춰 한 번만 반환합니다.
5. 담당 모듈이 대상·날짜·URL·상태 전이를 검증하고 저장 또는 조회를 수행합니다.
6. 해석 실패, 낮은 신뢰도, 모호한 대상은 데이터를 바꾸지 않고 짧게 다시 묻습니다.

```text
Telegram 메시지
  ├─ 고정 명령·버튼 ──────────────> 결정적 처리 (AI 0회)
  └─ 자유문장 + 제한된 답장 문맥 ─> 전역 구조화 해석 (AI 1회)
                                      └─ 모듈 검증 ─> 조회/변경
```

## AI와 코드의 책임

AI가 맡는 일:

- 사용 의도와 도메인 구분
- 목록 번호·회사명·제목 표현을 실제 답장 대상에 연결
- 일정·다음 행동·피드백 의미 등 구조화된 필드 추출
- 정보가 부족할 때 필요한 확인 질문 생성

코드가 맡는 일:

- 인증, 권한, 버튼 callback과 slash command 처리
- DB 조회·변경, 상태 전이, 중복 방지와 재시도
- 실제 날짜 존재 여부, 미래 시각, URL, 대상 ID 검증
- 영구 제외 규칙과 리마인더의 승인 경계
- 수집기에서 공식 사이트 응답·HTML·날짜·URL을 기계적으로 파싱하는 작업

코드에 남은 정규식은 `/remind YYYY-MM-DD HH:MM ...` 같은 명시적 프로토콜, URL·날짜·HTML 추출과 입력 검증용입니다. 사용자의 자연스러운 의도를 추측하는 문구 목록이나 의미 정규식 fallback은 두지 않습니다.

## 지원하는 의미 경로

| 영역 | 대표 동작 |
| --- | --- |
| 브리핑·추천 | 오늘 브리핑, 주택·채용 전체/다음 목록, 안 보이는 이유 설명 |
| Inbox | 저장, 목록, 상세 보기, 수정, 완료, 취소, 알림 제안 |
| 리마인더 | 생성, 부족한 시각 확인, 목록, 취소 |
| 피드백 | 좋음/싫음/지원, 취소, 기록, 영구 규칙 제안·조회·삭제 |
| 주택 | 지원 결과, 발표일, 지원 현황, 사용법 |
| 채용 | 지원 현황 |
| 운영 | 읽기 전용 `/ask` 질문 |

새 자유문장 기능을 추가할 때는 전역 스키마에 경로와 필드를 추가하고, 한 모듈에 실행 어댑터를 둡니다. 모듈 안에 별도의 AI 호출이나 의미 정규식을 추가하지 않습니다.

## 답장 문맥과 개인정보

전역 해석기는 답장한 메시지에서 최대 20개의 공개 라벨만 받습니다. 항목 번호, 제목, 회사, 출처, 도메인 정도이며 채용 공고 전문, 개인 프로필, 인증정보와 쿠키는 포함하지 않습니다. 사용자 문장과 라벨은 모두 신뢰할 수 없는 데이터로 취급하며, 해석 과정은 검색·파일 읽기·명령 실행을 할 수 없습니다.

## 변경 안전장치

- 대상이 여러 개로 해석되면 번호나 제목을 한 번 더 묻습니다.
- 불가능한 날짜나 과거 알림 시각은 저장하지 않습니다.
- AI 호출이 실패하면 정규식으로 대신 추측하지 않습니다.
- 일반 피드백은 해당 항목의 선호 기록이고 영구 제외가 아닙니다.
- 앞으로도 적용되는 제외 규칙과 정시 알림은 사용자 확인 후 활성화합니다.
- 동일 Telegram update가 재처리되어도 안정적인 source key로 중복 변경을 막습니다.
- 공고 제외 이유는 AI가 만들지 않고 현재 DB와 추천 스냅샷을 비교해 결정합니다.

## 비용과 지연 원칙

- 자유문장: 전역 구조화 AI 호출 1회
- 고정 slash command와 버튼: AI 호출 0회
- 주택 평가, 채용 필터, Wanted 검색 같은 백그라운드 AI 작업은 메시지 해석과 별도
- 도메인 모듈의 2차 해석 호출은 금지해 호출 수와 의미 불일치를 줄임

운영 계정의 Codex 사용량·인증 오류 시 API fallback은 별도 예산 제한을 따르며, 해석 중간에 임의로 쓰기 권한이나 도구가 추가되지 않습니다.

## 예시

| 사용자 표현 | 구조화 결과 예시 |
| --- | --- |
| `채용 공고 싹 보여줘` | `recommendations_list / jobs` |
| `큐픽스 공고 왜 안 보여?` | `recommendation_explain / jobs / title=큐픽스` |
| 채용 목록에 `두 번째 지원했어` | `feedback / jobs / target 2 / applied` |
| 주택 목록에 `이런 민간임대는 앞으로 빼` | `preference_rule / housing` 후 적용 확인 |
| Inbox 목록에 `3번 다음 주 화요일로 바꿔` | `inbox_update / target 3 / event_at` |
| `내일 병원 알려줘` | 시간이 없어 `reminder_clarify` |

사용자 관점의 짧은 사용법은 [TELEGRAM-MANUAL.md](./TELEGRAM-MANUAL.md), 전체 컴포넌트 경계는 [../ARCHITECTURE.md](../ARCHITECTURE.md)를 참고합니다.
