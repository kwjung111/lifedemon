# Telegram 메시지 해석 설계

Life Daemon은 자유문장을 기능별 정규식으로 흉내 내지 않습니다. Telegram 자유문장은 먼저 전역 라우터가 구조화합니다. 일반 기능은 담당 모듈이 결과를 검증해 실행하고, 추천 피드백은 제한된 도구 호출 에이전트가 필요한 조회와 변경을 여러 단계로 수행합니다.

## 한 메시지의 처리 순서

1. 런타임이 개인 채팅 ID와 사용자 ID를 확인합니다.
2. 고정 명령어와 버튼 콜백이면 AI 없이 정해진 규약으로 처리합니다.
3. 그 밖의 자유문장은 답장한 메시지의 공개 항목 정보와 함께 전역 AI 해석기로 보냅니다.
4. 라우터는 `route`, `domain`, 신뢰도와 기능별 필드를 JSON 스키마에 맞춰 반환합니다.
5. 일반 기능은 담당 모듈이 대상·날짜·URL·상태 전이를 검증하고 저장 또는 조회를 수행합니다.
6. 추천 피드백은 에이전트가 최대 세 라운드 동안 공고 상세·피드백 이력을 조회하고, 검증된 저장·지원·되돌리기 도구를 최대 열 번 호출합니다.
7. 에이전트는 확실한 작업을 먼저 처리하고 실제 반영 결과를 함께 보여주며, 남은 모호함만 짧게 묻습니다.

```text
Telegram 메시지
  ├─ 고정 명령·버튼 ──────────────> 결정적 처리 (AI 0회)
  └─ 자유문장 + 제한된 답장 문맥 ─> 전역 구조화 라우팅
                                      ├─ 일반 기능 ─> 모듈 검증 ─> 조회/변경
                                      └─ 추천 피드백 ─> 에이전트 ↔ 허용 도구 ─> 최종 보고
```

## AI와 코드의 책임

AI가 맡는 일:

- 사용 의도와 도메인 구분
- 일반 기능의 목록 번호·회사명·제목 표현을 실제 답장 대상에 연결
- 일정·다음 행동·피드백 의미 등 구조화된 필드 추출
- 정보가 부족할 때 필요한 확인 질문 생성

추천 피드백 에이전트가 맡는 일:

- 사용자 목표를 여러 항목과 여러 행동으로 분해
- 필요한 공고 상세와 기존 피드백 이력을 선택적으로 조회
- 조회 결과를 본 뒤 다음 도구를 고르는 적응형 실행
- 검증된 실제 반영 결과와 남은 질문을 자연어로 보고

코드가 맡는 일:

- 인증, 권한, 버튼 콜백과 슬래시 명령 처리
- DB 조회·변경, 상태 전이, 중복 방지와 재시도
- 날짜 형식, 미래 시각, URL, 대상 ID 검증
- 영구 제외 규칙과 리마인더의 승인 경계
- 수집기에서 공식 사이트 응답·HTML·날짜·URL을 기계적으로 파싱하는 작업

코드에 남은 정규식은 `/remind YYYY-MM-DD HH:MM ...` 같은 명시적 규약, URL·날짜·HTML 추출과 입력 검증용입니다. 사용자의 자연스러운 의도를 추측하는 문구 목록이나 의미 정규식 대체 경로는 두지 않습니다.

## 지원하는 의미 경로

| 영역 | 대표 동작 |
| --- | --- |
| 브리핑·추천 | 오늘 브리핑, 주택·채용 전체/다음 목록, 안 보이는 이유 설명 |
| Inbox | 저장, 목록, 상세 보기, 수정, 완료, 취소, 알림 제안 |
| 리마인더 | 생성, 부족한 시각 확인, 목록, 취소 |
| 피드백 | 좋음/싫음/지원, 취소, 기록, 영구 규칙 제안·조회·삭제 |
| 주택 | 지원 결과, 발표일, 지원 현황, 사용법 |
| 채용 | 지원 현황 |
| 운영 | 읽기 전용 `/ask` 질문 |

새 자유문장 기능을 추가할 때는 기본적으로 전역 스키마에 경로와 필드를 추가하고 한 모듈에 실행 어댑터를 둡니다. 여러 조회 결과에 따라 다음 행동이 달라지는 기능만 명시적인 에이전트 경로와 좁은 도구 집합을 사용합니다. 의미 정규식 대체 경로는 추가하지 않습니다.

## 답장 문맥과 개인정보

전역 라우터는 답장한 메시지에서 최대 20개의 공개 라벨만 받습니다. 항목 번호, 제목, 회사, 출처, 도메인과 짧은 공개 판정 요약 정도이며 개인 프로필, 인증정보와 쿠키는 포함하지 않습니다. 피드백 에이전트는 사용자가 언급한 대상을 판단하는 데 필요할 때만 제한된 공고 내용을 조회할 수 있습니다. 사용자 문장, 라벨, 공고 내용과 도구 결과는 모두 신뢰할 수 없는 데이터로 취급하며 셸·파일·네트워크·임의 SQL 도구는 제공하지 않습니다.

## 변경 안전장치

- 한 문장에 여러 대상이 있으면 에이전트가 각각 독립된 도구 호출로 반영합니다. 확실한 대상과 불명확한 대상이 섞이면 확실한 행동부터 실행하고 불명확한 부분만 묻습니다.
- 실행기는 답장 문맥에 없는 항목 ID와 잘못된 도구 인자를 거부합니다.
- 불가능한 날짜나 과거 알림 시각은 저장하지 않습니다.
- AI 호출이 실패하면 정규식으로 대신 추측하지 않습니다.
- 일반 피드백은 해당 항목의 선호 기록이고 영구 제외가 아닙니다.
- 앞으로도 적용되는 제외 규칙과 정시 알림은 사용자 확인 후 활성화합니다.
- 출처 키를 사용하는 변경은 동일한 Telegram 갱신이 재처리돼도 중복 저장을 막습니다.
- 공고 제외 이유는 AI가 만들지 않고 현재 DB와 추천 스냅샷을 비교해 결정합니다.

## 비용과 지연 원칙

- 일반 자유문장: 전역 구조화 AI 라우팅 1회
- 추천 피드백: 전역 라우팅 후 보통 에이전트 판단 2회, 상세 확인이 필요하면 최대 4회
- 고정 슬래시 명령과 버튼: AI 호출 0회
- 주택 평가, 채용 필터, Wanted 검색 같은 백그라운드 AI 작업은 메시지 해석과 별도
- 에이전트는 최대 세 판단 라운드·열 도구 호출로 제한하며 같은 호출을 반복하지 않음

운영 계정의 Codex 사용량·인증 오류 시 API 대체 경로는 별도 예산 제한을 따르며, 해석 중간에 임의로 쓰기 권한이나 도구가 추가되지 않습니다.

## 예시

| 사용자 표현 | 구조화 결과 예시 |
| --- | --- |
| `채용 공고 싹 보여줘` | `recommendations_list / jobs` |
| `A회사 공고 왜 안 보여?` | `recommendation_explain / jobs / title=A회사` |
| 채용 목록에 `두 번째 지원했어` | `feedback / jobs / target 2 / applied` |
| 통합 목록에 `2번은 별로고 5번은 좋아` | `feedback / mixed` → 에이전트가 두 저장 도구 실행 |
| 주택 목록에 `이런 민간임대는 앞으로 빼` | `preference_rule / housing` 후 적용 확인 |
| Inbox 목록에 `3번 다음 주 화요일로 바꿔` | `inbox_update / target 3 / event_at` |
| `내일 병원 알려줘` | 시간이 없어 `reminder_clarify` |

사용자 관점의 짧은 사용법은 [TELEGRAM-MANUAL.md](./TELEGRAM-MANUAL.md), 전체 컴포넌트 경계는 [../ARCHITECTURE.md](../ARCHITECTURE.md)를 참고합니다.
